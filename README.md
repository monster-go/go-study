## 新人如何快速掌握 Go 知识点

Go 的学习曲线并不陡峭，但容易在「语法会了、工程不会」或「命令会用、原理不懂」之间卡住。下面是一套面向新人的学习路径，也是本仓库**后续所有知识点文档必须遵循的编写要求**。

### 一、学习路径：先跑通，再深入

按以下顺序推进，每完成一步再进入下一步，避免一上来啃语言细节：

| 阶段 | 目标 | 对应文档 / 示例 |
|------|------|-----------------|
| 1. 环境跑通 | 能安装、验证、运行第一个程序；熟悉 go 命令 | [安装与入门](docs/go-install-and-getting-started.md) → `example/hello-mod/`；[工具链与命令](docs/go-toolchain-and-commands.md) |
| 2. 工程认知 | 理解模块、构建、依赖怎么管理 | [构建与依赖演进](docs/go-build-and-dependency-evolution.md) → `example/calc/` |
| 3. 语法基础 | 变量、类型、控制流、函数、结构体 | [变量与常量](docs/go-variables-types-and-constants.md) → `example/vars/`；[fmt 包](docs/go-fmt-package.md) → `example/fmt/`；[运算符](docs/go-operators.md) → `example/operators/`；[字符串](docs/go-string-operations.md) → `example/strings/`；[语句控制](docs/go-control-flow.md) → `example/controlflow/`；[数组、切片与 map](docs/go-arrays-and-slices.md) → `example/slices/`、`example/maps/`（函数、结构体待写） |
| 4. 核心特性 | 接口、错误处理、并发（goroutine / channel） | 待写 |
| 5. 标准库与生态 | `fmt`、`net/http`、测试、常用第三方库 | [fmt 包](docs/go-fmt-package.md) → `example/fmt/`（其余待写） |
| 6. 工程实践 | 项目布局、日志、配置、部署 | 待写 |

**原则：每一篇文档都要能独立动手验证**，不能只讲概念不给可运行的代码。

### 二、知识点文档编写要求

本仓库每篇知识点文档须满足以下规范，以保证新人「读得懂、做得出、查得到」。

#### 1. 文档结构（固定模板）

每篇文档必须包含以下部分，顺序不可打乱：

```markdown
# 标题：副标题（说明本篇解决什么问题）

> 知识点总结：用 1～2 句话概括读者读完能做什么。

---

## 1. 为什么需要了解这个

（动机：解决什么痛点、和前后知识点什么关系、常见误区是什么）

## 2. 核心概念

（定义 + 对比表 + 图示/目录结构，先建立心智模型）

## 3. 动手实践

（完整命令或可运行代码，逐步操作，每步说明预期输出）

## 4. 常见坑与排查

（至少 2 条真实易错点 + 如何验证/修复）

## 5. 小结与延伸阅读

（3～5 条要点回顾 + 官方文档链接）
```

#### 2. 内容原则

| 要求 | 说明 |
|------|------|
| **先 Why 后 How** | 每节开头用一两句说明「为什么要学这个」，再讲怎么做 |
| **一知识点一文档** | 单篇聚焦一个主题，篇幅控制在 300～800 行；过长则拆分 |
| **可运行优先** | 所有代码示例必须能在 `example/` 下找到对应目录，或文档内给出完整可复制片段 |
| **对比优于罗列** | 有历史演进、多种写法、易混淆概念时，用表格对比（参考已有文档风格） |
| **命令带注释** | Shell 示例每行或每组命令附简短注释，说明在做什么、预期结果是什么 |
| **术语中英对照** | 首次出现的关键术语标注英文，如「模块（Module）」「工作区（Workspace）」 |
| **版本说明** | 涉及工具链行为时注明适用的 Go 版本（如 `go 1.21+`） |
| **链接官方源** | 延伸阅读优先引用 [go.dev](https://go.dev) 官方文档，而非二手博客 |
| **把原理讲清楚** | 不只给出「能不能做」的结论，还要说明**语言或机制层面的原因**，让读者理解后能自行推导同类规则，而不必死记 |
| **示例可推导（不跳步）** | 关键示例不能只写「输入 → 输出」的结论链；须交代**中间步骤**（换算、逐位运算、类型转换等），让读者能自行复算，而非对着黑箱等式死记 |

**「把原理讲清楚」示例：** 讲 slice / map 不能用 `==` 比较内容时，应像 [运算符文档](docs/go-operators.md) 那样点明——它们是**引用类型**，`==` 比较的是**是否指向同一块底层数据**，而非内容是否相同；因此编译器禁止对 slice 做 `==`，map 同理，且它们**只能与 `nil` 比较**。先讲清「为什么不行」，再给出 `slices.Equal`、`reflect.DeepEqual` 等替代写法。

**反面写法（避免）——只给结论、不讲原因：** 仅写「slice 和 map 不能 `s1 == s2`，请用 `reflect.DeepEqual`」——读者知道怎么做，却不知道原因，遇到类似规则（如 function 只能 `== nil`）时仍无法举一反三。

**反面写法（避免）——跳步示例、推导过程缺失：** 位运算只写 `12 & 10` → `1100 & 1010` → `1000` → `8`，却不说明 12/10 如何写成二进制、每一位如何与/或/异或、结果二进制如何换回十进制——读者看到等式成立，却不知道等式怎么来的。此类遗漏与「把原理讲清楚」互补：前者缺**机制原因**，后者缺**可复现的推导步骤**；涉及进制换算、位运算、浮点精度、内存布局等时，两者往往都要写。

#### 3. 示例代码要求

- 每个独立主题在 `example/<主题名>/` 下提供最小可运行示例，包含 `go.mod`（如适用）
- 示例命名见名知意：`hello-mod`、`calc`，避免 `test1`、`demo`
- 示例代码保持简短（通常单个文件 < 50 行），注释只解释非显而易见的逻辑
- 文档中引用示例时使用相对路径，并说明运行方式：

  ```bash
  cd example/hello-mod
  go run .
  ```

#### 4. 面向新人的表达规范

- **句子短、段落短**：一段不超过 5 行，避免大段 prose
- **避免 assumed knowledge**：不默认读者懂 C/Java/操作系统细节；必要时用一句话类比
- **误区要提前说**：网上过时说法（如「项目必须放 GOPATH」）在文档靠前位置主动澄清
- **检查清单**：实践章节末尾给出「自检清单」，例如：
  - [ ] `go version` 输出符合预期
  - [ ] 示例程序能无报错运行
  - [ ] 能用自己的话解释本篇 3 个核心概念

#### 5. 文档索引与维护

- 每新增一篇文档，在本文「文档」章节追加链接，并更新上表「学习路径」中的「待写」项
- 文件名使用小写英文与连字符：`docs/go-concurrency-basics.md`
- 标题使用中文 + 冒号副标题，与现有两篇保持一致
- 文档间互相引用时使用相对路径，形成可跳转的知识网

### 三、新人自检：我是否真的掌握了？

每学完一篇，用下面四个问题自测；有四个「是」才算过关：

1. **能复述**：不用看文档，用自己的话说明这篇在讲什么、为什么重要？
2. **能操作**：不看文档，独立完成文档里的实践步骤？
3. **能排错**：故意改错一个配置或代码，知道报错原因并修回来？
4. **能迁移**：能把本篇知识用到一个小改动或迷你练习里？

若某一题答「否」，回到该文档的「常见坑与排查」和 `example/` 示例重做一遍，不要急着进入下一篇。

### 四、推荐学习节奏

| 时间 | 动作 |
|------|------|
| 每天 30～60 分钟 | 读一篇文档 + 跑通示例 + 完成自检 |
| 每周 | 回顾本周文档的小结，写 5 条自己的笔记（可放在各文档末尾或单独笔记） |
| 遇到卡住 | 先查文档「常见坑」→ 再查 [Go 官方文档](https://go.dev/doc/) → 最后再搜 Issue / 社区 |

坚持「读 → 写 → 跑 → 错 → 改」的循环，比一次性通读所有语法更快建立工程直觉。本仓库的文档会按上述要求持续补充，欢迎按同一规范贡献新的知识点。
